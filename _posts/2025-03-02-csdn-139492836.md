就是自动化生成大量样本并通过将样本通过某种方法变异输入给测试程序，根据程序的反馈得到该程序是否存在相关漏洞

## 两种主流fuzz的基本原理
- 基于生成输入数据fuzz：1.随机  2. 基于模板生成（如HTTP报文  里面各个字段又可分为 固定不变 任意变动 有限变动）
- 基于变异-覆盖制导fuzz：样本变异->触发新路径，感知进程反馈->基于反馈 变异样本再输入


# AFL原理


## 路径信息反馈
反馈流程：
	1. 数据生成器
	2. 数据输入
	3. 执行进程（插桩了的）
	4. 路径信息统计规整
	5. 反馈给数据生成器做出针对性变异

如何插桩：
1. 汇编层：汇编语言到二进制的过程中寻找条件跳转点比如JNZ adder，插入函数， 不同汇编架构插装的函数汇编不同
2. llvm pass（官方api）：clang/llvm  IR不同架构通用，在IR这里插桩
  clang：高级语言->IR
  llvm:IR->汇编/二进制
  
插桩功能：覆盖信息反馈

    1. 启动fuzz时候，会有afl-fuzz主进程，它会开辟一片共享内存，并执行程序进程. 
    2. 程序进程执行时候触发不同路径会执行afl-maybe-log(random_num 参数是插桩时随机生成的一个数),为当前路径生成哈希索引 
    3. 进程将共享内存的哈希索引位置的值+1 . 
    4. afl-fuzz访问共享内存，知道路径信息
    



 代码层:计算当前索引
   

```cpp

   afl_pre_loc=random_num>>1(防止A到A和B到B都是异或为0)
```

## forkserver


alf-fuzz第一次被执行 程序进程使得程序进程变成forkserver

```cpp

if ( !_afl_area_ptr ) // fork_server_is_on
{

    if ( _afl_global_area_ptr )
    {
        afl_area_ptr = _afl_global_area_ptr;
    }
    else  //这里
    {
        v17 = v4;
        v18 = random_num;
        afl_shm_id_str = getenv("__AFL_SHM_ID"); // afl-fuzz通过环境变量AFL_SHM_ID传递共享内存handle
        if ( !afl_shm_id_str
            || (v11 = atoi(afl_shm_id_str), afl_shm_addr = shmat(v11, 0LL, 0), afl_shm_addr == (_BYTE *)-1LL) )
        {
            ++_afl_setup_failure;
            v5 = v17;
            return v5 + 127;
        }
        *afl_shm_addr = 1;
        _afl_area_ptr = (__int64)afl_shm_addr; // target
        _afl_global_area_ptr = afl_shm_addr;
        v16 = afl_shm_addr;
        if ( write(199, &_afl_temp, 4uLL) == 4 ) // 通过199管道通知afl-fuzz, forkserver已准备完毕
        {
            while ( 1 ) // 开始作为forkserver
            {
                v13 = 198;
                if ( read(198, &_afl_temp, 4uLL) != 4 ) // 收到afl-fuzz指令，fork新进程
                    break; // 如果198管道关闭，forkserver退出
                LODWORD(v14) = fork();
                if ( v14 < 0 )
                    break;
                if ( !v14 ) // 父进程(forkserver)，子进程是目标程序进入afl_fork_resume，关闭由forkserver继承而来的管道后，正常执行
                    goto _afl_fork_resume;
                _afl_fork_pid = v14;
                write(199, &_afl_fork_pid, 4uLL); //传输子进程pid给afl-fuzz
                v13 = _afl_fork_pid;
                LODWORD(v15) = waitpid(_afl_fork_pid, &_afl_temp, 0); // 父进程(forkserver)获得子进程pid，并通过waitpid等待子进程退出
                if ( v15 <= 0 )
                    break;
                write(199, &_afl_temp, 4uLL); // afl_temp保存子进程退出信息，由199管道传递给afl-fuzz后继续循环等待下一次指令
            }
        }
    }
}
```

##  样本变异
[https://rk700.github.io/2018/01/04/afl-mutations/](https://rk700.github.io/2018/01/04/afl-mutations/)
得到样本的路径信息后会进行评估，合适就会将样本加入待变异队列里，这个样本接下来会继续做变异
确定性变异：
- bitflip：按位翻转
路径没变化，认为是无价值字段
路径变化相同，认为是固定组合token
- arithmetic：整数加/减算术运算 
- interest：替换特殊数值(临界值)
- dictionary：把自动生成（位翻转）或用户提供的token替换/插入到原样本
不确定性变异：
- havoc：随机变异，随机选一种变异方法
- splice：样本拼接

## 覆盖制导的问题

针对大型项目：
1.hash简单（异或生成索引）
2. 共享内存对应的数组长度有限
3. 路径过多会导致哈希冲突（多条路径映射到同一个位置 AFL通过路径过多时减少插桩数目但并没有实质解决问题）

反制：afl-fuzz对目标程序的第一次执行会变成forkserver，然后fork子进程
4. 检测afl-fuzz特征（如共享内存环境变量），清空或者直接写共享内存，扰乱fuzz
5. 检测198/199管道存在或者直接读写该管道则可以判断存在AFL在fuzz

# ASAN
## 开源ASAN
检测内存访问错误的工具

编译开启标志 -fsanitize=address  ，执行程序出现内存访问错误时会崩溃

- 不可访问如UAF  double free
- 越界读写问题

KASAN和ASAN本质差不多，都是由编译器编译决定的

将数据区域分为可访问区域和不可访问区域（red_zone）

影子内存和正常内存的比例是1:8，代表一个字节表示八个字节（对齐的）的内存访问信息，二者存在映射关系且是分别一块区域（一块区域是影子内存 一般固定是0x7FFF8000）

如内核中
![在这里插入图片描述](/images/csdn/2025/csdn-139492836/image-005-02df10a672.png)

 影子内存值：1-7代表多少个字节可访问 0为都能访问，负数为都不可访问

检测数据区域是否可访问，大致如下，详细可参考[linux内核（5.4.81）——KASAN](https://www.anquanke.com/post/id/237502)

__asan_loadN(addr)/__asan_storeN(addr)主要功能：
1. 检测 要检测的地址是否在影子内存映射的区域内
2. 然后检测addr所在位置的N bytes内存是否可用


 设置数据区域是否可访问
 - 堆 不可访问如UAF  double free
   malloc时设置对应影子内存区域为0
   free时设置对应影子区域为负数或者别的（因为可能库函数里要用到被释放的区域）
-  栈 越界读写问题
  两个数组连续问题，会导致检查不出来越界，因为越界到下一个数组显示可访问
  函数开始处，开辟空间时候会在数组前后都分别设置一块区域对应影子内存为不可访问（负数），对数组对应影子内存区域设置为0


IDA中发现堆相关设置操作集成到了malloc和free里面，当访问堆上内存会有判断对应影子内存值为0或者大于此时正访问的size

IDA发现栈开辟了一个更大的空间，并且包含asan描述信息和redzone和数组，并设置了影子内存值

ASAN的设置区域和检测区域主要由插桩实现

 
对于闭源，通过二进制程序得到汇编之后，再对代码段进行插桩

因为是要对函数的起始和终止处进行插桩，如设置某些区域，

如何找？
1. 代码段所有指令都插桩，不断丢弃尝试，直到插桩正常，没有崩溃。目的是插桩所有函数的起点
2. 函数表来直接插桩

不破坏原功能情况插入代码存在问题
1. 插入汇编代码可能会导致汇编代码位置改变，如跳转指令可能跳转地址不变，但对应内容改变了
2. inline hook: 插入跳转指令（指令长度小 优于直接插 但也会有第一种方案的后果），跳转指令跳转的位置是插桩函数，然后再回来

中间语言插桩，不通过汇编插，因为不同架构比较麻烦，二进制->汇编->IR，对IR插桩，平滑各种架构

寻找汇编语言编译器

### edit_params

1. 根据命令行参数寻找对应的c语言编译器
