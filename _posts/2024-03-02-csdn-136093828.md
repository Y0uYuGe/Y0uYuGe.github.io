# GBN协议的弊端
批量重传导致多余的帧也要重传

设置单个确认就是可以逐一对帧确认
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-001-a2d3680dca.png)
# 选择重传协议中的滑动窗口
发送方：窗口内如果下界没有接收到确认和其他部分接收到确认，那么窗口将直到下界接收到确认才会一起移动相应的窗口数

接收方：下界没有接收到，但是其相邻的右边那个接收到了，那么此时如果下界接收到，将会移动两个窗口
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-002-f60a30e221.png)
# SR发送方必须响应的三件事
如果窗口移动了并且有序号在窗口内的未发送帧，则发送这些帧：就是窗口可能移动后，现在窗口内有未发送帧了，那么就会发送这些帧
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-003-550f775326.png)
# SR接收方要做的事

直到所有帧（即序号更小的帧）皆被收到为止：即当收到6，7后不会移动窗口，直到收到5才会移动窗口
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-004-4028069175.png)
# 运行中的SR
接收方即使接收到3，4，但没有接收到2也不会右移窗口
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-005-a17721f9fb.png)
# 滑动窗口长度
大了：发送窗口大于接收窗口，接收不过来
小了：浪费多于的接收窗口

此时左边的0号为旧帧
右边的0号为新帧，但此时接收方是无法区分是新帧还是旧帧的
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-006-20f0989313.png)
# SR协议重点总结
收一个确认一个是局限于接收窗口内的
![在这里插入图片描述](/images/csdn/2024/csdn-136093828/image-007-46bbf73f75.png)
# 习题1
不存在累积确认：即收到1号确认并不代表0号也被确认了，SR协议每个都需要单独确认
